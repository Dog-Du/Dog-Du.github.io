# AGENTS 工作流生成器

## 概述

为新主题生成 `AGENTS-learning-&lt;topic&gt;.md` 长期学习工作流文件。根据学习类型自动选择合适的事实标准、文章结构和章节模板。

**核心原则：所有 AGENTS 文件共享同一套骨架（目标 → 核心目标 → 事实标准 → 产物规则 → 文件职责 → 恢复流程 → 索引 → 推进判断 → 文章结构 → 主线 → 问题记录 → 分支指南 → 操作示例 → 完成说明 → 边界 → 版本），差异仅存在于三个类型相关节。**

## 何时使用

- 用户说"帮我为 XXX 创建学习框架"、"仿照 learn-vim 的方式为 XXX 搭工作流"、"我要系统学习 XXX"
- 需要将一次性调研需求转化为结构化、可恢复的长期学习系统
- 目标主题需要按天/专题推进，产出可回看的 Markdown 文章

**不使用的情况：**
- 一次性问答，不需要长期追踪进度
- 只是查某个具体 API/命令的用法

## 学习类型分类

生成前先判断属于哪种类型。决定权在 Section 2（事实标准/学习方式）和 Section 8（文章结构）。

| 类型 | 适用场景 | 事实标准 | 文章风格 | 参考案例 |
|------|---------|---------|---------|---------|
| **A — 工具/技能** | Vim、Git、编程语言、框架 | 本地实际行为（--version、--help、实际命令输出） | 任务驱动 + 概念骨架 | `learn-vim` |
| **B — 源码/技术深挖** | RocksDB、Redis、内核子系统 | 本地源码 clone(path\to\\&lt;repo&gt;） | 源码驱动，函数名作证据锚点 | `learn-rocksdb` |
| **C — 行业/领域调研** | 数据库行业、AI 全景、云原生 | 公开资料（官方文档、论文、博客）+ 可获取时源码验证 | 调研报告式，对比框架 | `learn-database-industry` |

### Type A 特征
- 无源码锚点
- 可以有每日 5-10 分钟练习、环境差异小节
- 小节 7（推进判断）：5 条场景化检查（能否独立完成练习、能否说清核心套路等）

### Type B 特征
- 有源码路径、调用链追踪、Mermaid 机制图
- 可选模块：设计动机、横向对比、工程启发
- 小节 7（推进判断）：5 条源码化检查（能否指出核心文件、能否讲清主流程等）
- 额外包含"优先关注的源码目录"和"复习问答闸门"

### Type C 特征
- "专题"而非"天"是推进单位
- 有独立单系统模板（比如 17 项）和专题预览模板（比如 8 项）
- 有领域专属分析框架（如 Storage-First 权重）、引用优先级规则、插件/生态分析规则
- 额外包含"调研某个系统"执行分支

## 生成流程

### 第一步：收集参数

向用户确认（或从上下文推断）以下参数。缺省值用方括号标注：

| 参数 | 说明 | 示例 |
|------|------|------|
| 主题名 | 中文显示名 | "Git 版本控制" |
| 英文短名 | 用于文件命名 | `git` |
| 学习类型 | A / B / C | A |
| 输出路径 | [content/posts] | `content/posts` |
| 文章前缀 | [learning-&lt;en&gt;-day] | `learning-git-day` |
| 索引文件 | [&lt;prefix&gt;-day000-index.md] | — |
| 本地源码路径 | Type B 必填 | `D:\\program\\redis` |
| 搜索关键词 | 恢复扫描用，3-6 个 | `git`、`version control`、`GitHub` |
| 学习主线 | 用户规划的章节序列 | 见下文 |

### 第二步：按类型生成文件

按选定的类型（A/B/C），生成完整的 `AGENTS-learning-&lt;topic&gt;.md`，写入 `something-prompt/learn-&lt;topic&gt;/`。所有 14 个通用节全部包含，只替换类型相关占位符。

通用骨架见下方"快速参考"表。类型相关节的内容规则见下文"类型相关节"。

### 第三步：检查

生成后自检：
- [ ] "目标"节是否包含类型特有的一句话
- [ ] Section 2 是否匹配类型（事实标准/学习方式）
- [ ] Section 7（推进判断）是否使用了类型正确的检查项
- [ ] Section 8（文章结构）是否匹配类型的模板
- [ ] Type C 是否有额外的单系统模板、引用规则、插件分析、专题序列
- [ ] 所有 `&lt;...&gt;` 占位符是否已替换为具体值

## 通用骨架速查

以下 14 节在所有类型的 AGENTS 文件中都存在（Type C 可能有额外的专属节）。`&lt;...&gt;` 为需替换占位符。

| 节号 | 节名 | 内容要点 | 类型差异 |
|------|------|---------|---------|
| — | 目标 | 四句话：恢复进度 → 按意图执行 → 持久化到 content/posts → 类型特有目标 | 类型 A：学习主线路径；类型 B：本地源码是事实标准；类型 C：storage-first 视角 |
| 1 | 核心目标 | 建立长期可恢复学习系统；每天一篇可回看文章；类型特有文章风格 | 类型 A：任务驱动+概念骨架；类型 B：源码驱动的学习型文章；类型 C：调研报告式个人学习文档 |
| 2 | 事实标准/学习方式 | **类型决定本节内容** | 见下方"类型相关节" |
| 3 | 产物目录与命名规则 | 输出目录（content/posts）；命名格式（prefix-dayXXX-YYYY-MM-DD-topic.md）；索引固定名；Day 000 特例 | 无差异 |
| 4 | 文件职责划分 | 每日文章是主载体；索引文件只做导航和轻量状态维护 | 无差异 |
| 5 | Session 恢复流程 | 扫描 day 文章 → 扫描关键词历史 → 读索引 → 恢复状态 → 类型特有验证 → 按意图分支 | 步骤 5（验证）：Type A 检查本地命令/版本；Type B 检查本地源码目录存在性；Type C 检查本专题所需源码 |
| 6 | 索引文件记录内容 | 进度、主题、阶段、文章索引、薄弱点、下一步建议；mastery_score（0-5）+ understanding_status（green/yellow/red） | 无差异 |
| 7 | 推进判断 | "理解约 80% 即可推进"；5 条类型特有检查；green/yellow/red 判定 | 见下方"类型相关节" |
| 8 | 每日文章标准结构 | 15-16 项文章模板；front matter 要求；风格规则；Mermaid 原则 | 见下方"类型相关节" |
| 9 | 默认学习主线 | 用户提供的章节/专题序列 | 类型 A/B：按天编号序列；类型 C：按阶段/专题表格 |
| 10 | 问题记录规则 | 问题和外部问题写入当天文章；分为"我的问题"和"外部高价值问题"；每题写：问题、简答、依据、结论、是否回看 | 依据字段：Type A "场景"，Type B "源码依据"，Type C "来源/源码依据" |
| 11 | 执行分支指南 | 继续学习 / 提问 / [Type C: 调研某个系统] | 继续学习的内容规则按类型变化 |
| 12 | 常用操作示例 | 开始 / 继续 / 指定主题 / 问问题 / 更新索引 | Type B 额外："做专题"；Type C 额外：指定专题跳转 |
| 13 | 完成后必须说明 | 进度、位置、文件变更、参考资料类型、下一步 | 参考类型：Type A "帮助/文档/命令"；Type B "源码目录/文件"；Type C "来源/源码目录" |
| 14 | 重要边界 | 不假装记住进度；索引不长篇；类型特有边界 | Type A：不写命令词典、低频不放主线；Type B：不堆函数名清单；Type C：不写成产品百科 |
| — | 版本 | 唯一权威定义声明 | 无差异 |

## 类型相关节

### Section 2（事实标准 / 学习方式）

**Type A — 事实标准与资料优先级**：三级体系

一级：本地实际行为（--version、--help、实际命令输出）。说明"XX 学习不以源码为事实标准，最重要是实际装了什么版本"。
二级：官方文档。
三级：社区资料（辅助理解，非最终标准）。
约束：优先讲怎么用/什么时候用/为什么顺手；未在本地验证的说法必须标注。

**Type B — 事实标准与资料优先级**：本地源码为主

一级：本地源码（固定路径 D:\program\&lt;repo&gt;），是权威真相源。
外部资料分三级：官方仓库内（Wiki/Issue/Discussion/PR）→ 社区（Stack Overflow/知乎/博客）→ 二手资料（仅线索）。
约束：能指源码位置不讲抽象概念；能说调用链不讲教科书定义；结构推断必须标注。

**Type C — 学习方式**：专题推进架构

专题是阶段分组，系统是学习单位，Day 是单篇文章编号。Topic 1 按后续专题展开（开篇→N 篇预览→收束）。Topic 2+ 按系统展开（开篇→N 篇系统文章→收束）。
专题不等于文章。Day 必须对应一篇实际文章。

### Section 7（推进判断）

**Type A** — 5 条场景化检查：能否说出解决什么实际问题 / 能否独立完成练习 / 能否说出核心操作套路 / 能否说明使用场景 / 能否说出一个不确定的问题。

**Type B** — 5 条源码化检查：能否指出核心源码文件 / 能否讲清主流程 / 能否说出 2-3 个关键数据结构 / 能否回答复习题 / 能否说出一个不确定的问题。

**Type C** — 不使用本节的 5 条检查格式。Type C 通过单系统文章深度规则中的"最低完成标准"来判断（读者能否看懂关键状态对象、沿图解释路径、说清 5 个 badcase、区分文档级与源码级结论）。

### Section 8（文章结构模板）

**Type A — 15 项模板**：今日主题、学习目标、前置回顾、典型场景、最小命令集/核心操作、它是怎么用的、常见操作套路、环境差异、今日练习（5-10分钟）、今日问题与讨论、常见误区或易混点、扩展内容、今日小结、明日衔接、复习题。

**Type B — 16 项模板**：今日主题、学习目标、前置回顾、源码入口、它解决什么问题、它是怎么工作的（含 Mermaid）、关键数据结构与实现点、源码细读、今日问题与讨论、常见误区或易混点、设计动机（可选）、横向对比（可选）、工程启发（可选）、今日小结、明日衔接、复习题。

**Type C — 不使用本节标准模板**。改为独立单系统模板（17 项：系统目标、workload、架构图、存储模型、写入/读取路径、日志恢复CDC、事务MVCC、复制一致性、元数据、二级索引、缓存后台任务、插件生态、我的问题、badcase、工程启发、引用来源）+ 专题预览模板（8 项）。

## Type C 额外专属节

Type C 的 AGENTS 文件除了 14 个通用节外，必须包含以下专属节（位于 Section 2 之后、Section 3 之前）：

| 节号 | 节名 | 内容 |
|------|------|------|
| 2.x | Domain-First 权重 | 将领域关注的模块分为高权重和低权重，如 storage-first 的"存储模型、写入路径、WAL、事务、复制、元数据、二级索引、缓存、后台任务"为高权重 |
| 2.x | 单系统文章深度规则 | 术语表要求、4-7 条核心机制路径、状态变化说明、模块级 badcase、Mermaid 图优先、源码函数名作证据锚点、最低完成标准 |
| 4.1 | 单系统学习模板 | 17 项完整模板（系统目标→workload→架构图→存储→写入→读取→日志→事务→复制→元数据→索引→缓存→插件→我的问题→badcase→工程启发→引用） |
| 4.2 | Topic 1 专题预览模板 | 8 项预览模板（为什么独立存在→代表系统→核心问题域→技术路线→badcase→工程启发→我的问题→引用） |
| 5.x | 资料来源与引用规则 | P0-P4 优先级表、开源/闭源系统源码规则、最低引用数量要求 |
| 6.x | 插件与生态补丁分析规则 | 四层分析（原生能力→官方扩展→外围组合→变通方案）、"能做"与"适合做"的区分 |
| 7.x | 默认专题序列 | 按阶段/专题/重点系统排列的表格 |

## 注入占位符速查

生成时将以下占位符替换为具体值：

| 占位符 | 替换为 |
|--------|--------|
| `&lt;topic-en&gt;` | 小写英文短名，如 `git` |
| `&lt;Topic&gt;` | 中文显示名，如 "Git 版本控制" |
| `&lt;content-path&gt;` | 输出路径，通常 `content/posts` |
| `&lt;prefix&gt;` | 文件前缀，如 `learning-git-day` |
| `&lt;search-keywords&gt;` | 3-6 个恢复扫描用关键词，用顿号分隔 |
| `&lt;local-source-path&gt;` | Type B 的本地源码路径 |
| `&lt;type-specific goal line&gt;` | Type A: 学习主线从 X 到 Y；Type B: 本地源码是事实标准；Type C: storage-first 视角 |
| `&lt;type-specific reference type&gt;` | Type A: 帮助、文档或本地命令；Type B: 源码目录或文件；Type C: 来源或源码目录 |
| `&lt;type-specific answer evidence&gt;` | Type A: 真实场景/命令/官方帮助；Type B: 本地源码+调用链；Type C: 公开资料/官方文档/本地源码 |

## 常见错误

1. **Type C 缺少专属节**：只写了 14 个通用节的骨架，漏掉了单系统模板、引用规则、插件分析等 Type C 必需节。
2. **Section 2 与类型不匹配**：Type A 的主题写了"本地源码是事实标准"；Type B 的主题写了"本地实际行为为准"。
3. **文章结构模板与类型绑定错误**：Type A 用了 16 项源码模板，Type B 用了 15 项任务模板。
4. **推进判断的 5 条检查项与类型不符**：Type A 问了"能否指出核心源码文件"，Type B 问了"能否独立完成练习"。
5. **占位符未替换**：生成的 AGENTS 文件中留下了 `&lt;topic-en&gt;`、`&lt;prefix&gt;` 等未替换标记。
6. **中文引号导致 Python 语法错误**：如果使用 Python 脚本生成文件，中文弯引号 “...” 在普通双引号字符串中会被误解析。改用 \u201c / \u201d 转义，或使用单引号包裹含弯引号的字符串。
