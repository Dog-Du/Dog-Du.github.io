# Vim 学习长期工作流规则

当用户要求在这个仓库中持续学习、讨论、整理、问答或持久化 Vim / Neovim / LazyVim 相关内容时，优先遵守本文件。

本文件定义的是长期固定规则，不是某一次 session 的临时任务说明。

## 1. 核心目标

- 目标是建立一套长期、连续、可恢复、可回看的 Vim 学习系统。
- 学习路径要从 `vim` 过渡到 `nvim`，最后稳定落到 `nvim + LazyVim`。
- 主线只覆盖日常最高频、最常用的内容，避免写成命令百科。
- 低频但有价值的内容要保留，但放在 `扩展内容` 中，作为后续回查资料。
- 每天的主产物是一篇可以长期积累和回看的 Markdown 学习文章。
- 写作风格采用：
  - 任务驱动为主
  - 概念骨架补充
  - 最终服务于真实日常编辑工作流

## 2. 事实标准与资料优先级

### 2.1 第一事实标准：本地实际行为

如果当前环境可用，优先以本地真实行为为准：

- `vim --version`
- `nvim --version`
- `vim` / `nvim` 中的 `:help`
- `nvim` 中的 `:checkhealth`
- 当前安装环境中的默认键位与实际行为

说明：

- Vim 学习不像 RocksDB 学习那样以源码为事实标准。
- 对 Vim 类工具，最重要的是：
  - 实际装了什么版本
  - 默认行为是什么
  - 当前环境里按键到底怎么工作

### 2.2 第二事实标准：官方文档

当需要补充说明或本地环境无法直接验证时，优先使用：

- Vim 官方帮助文档
- Neovim 官方帮助文档
- LazyVim 官方文档

这些来源优先于社区博客和二手总结。

### 2.3 第三优先级：高质量社区资料

社区资料可以作为问题来源和辅助解释来源，但不是最终事实标准。可用来源包括：

- Vim / Neovim 官方仓库 discussions 或 issue 中的说明
- 高质量博客
- 高质量教程
- Stack Overflow

使用这些资料时，最终结论仍应尽量回到：

- 本地实际行为
- 官方帮助文档
- LazyVim 官方文档

### 2.4 讲解时的约束

- 优先讲“怎么用、什么时候用、为什么这样更顺手”。
- 能用真实编辑场景解释时，不要只给命令清单。
- 如果某个说法没有在本地环境中直接验证，而是基于官方文档或经验推断，必须明确说明这一点。

## 3. 产物目录与命名规则

### 3.1 输出目录

- 所有 Vim 学习相关 Markdown 文件统一写入 `content/posts`。

### 3.2 文件命名

- 唯一索引文件固定为：
  - `content/posts/learning-vim-day000-index.md`
- 每日学习文章统一命名为：
  - `learning-vim-dayXXX-YYYY-MM-DD-<topic>.md`
- 其中：
  - `XXX` 为三位数字编号
  - `YYYY-MM-DD` 为本地日期
  - `<topic>` 为简洁英文短语，用连字符连接

### 3.3 Day 000 特例

允许存在一个 Day 000 主文档，作为学习总览与环境准备文章，例如：

- `learning-vim-day000-YYYY-MM-DD-overview-and-roadmap.md`

它和索引文件不同：

- Day 000 主文档承载知识正文
- 索引文件只负责导航和状态维护

## 4. 文件职责划分

### 4.1 每日文章是主载体

- 每日文章是知识主文档。
- 问题、回答、练习、误区、工作流建议，都优先写入当天文章。
- 每天尽量围绕一个主主题展开，必要时最多带一个副主题。
- 每篇文章都要独立可读，便于以后单篇回看。

### 4.2 索引文件是轻量状态文件

- 索引文件只负责：
  - 恢复进度
  - 导航每天文章
  - 记录掌握状态
  - 标记薄弱点和下一步建议
- 索引文件不承载长篇知识正文。
- 如果索引文件已存在，一律更新，不创建新的索引变体。

## 5. 新 Session 的恢复流程

每次开始新的 Vim 学习 session 时，必须按下面顺序恢复状态：

1. 扫描 `content/posts` 中所有 `learning-vim-day*.md`
2. 补充扫描其他标题、文件名或正文中包含：
   - `vim`
   - `nvim`
   - `neovim`
   - `lazyvim`
   的 Markdown，作为历史参考
3. 找到并读取唯一索引文件：
   - `content/posts/learning-vim-day000-index.md`
4. 恢复：
   - 当前是第几天
   - 已学过哪些主题
   - 上一篇文章写到哪里
   - 哪些章节是 `done`
   - 哪些章节是 `revisit`
   - 当前薄弱点是什么
5. 如果用户本次讨论涉及具体用法，优先检查：
   - 本地实际命令行为
   - `:help`
   - `vim --version`
   - `nvim --version`
   - LazyVim 官方默认工作流
6. 再根据用户本次意图继续学习或回答问题

## 6. 索引文件应该记录什么

索引文件应保持轻量但足够恢复上下文。

至少应记录：

- 当前学习总天数
- 当前最近一次学习主题
- 当前主线阶段
- 每日文章索引：
  - day 编号
  - 日期
  - 主题
  - 文件名
  - 状态
- 当前薄弱点
- 下一步建议

为了让新 session 不只知道“学过什么”，还知道“学得怎么样”，每一天还应记录：

- `understanding_status`
  - `green / yellow / red`
- `mastery_score`
  - `0-5`
- `weak_points`
- `source_anchors`

可选字段：

- `ready_for_next`
  - `yes / no`
- `next_review_trigger`

## 7. 如何判断一章是否可以推进

采用“理解约 80% 即可推进”的原则，但不能带着关键误解推进。

每天结束时至少检查：

1. 是否能说出这章解决了什么实际编辑问题
2. 是否能独立完成今天的 `5-10` 分钟练习
3. 是否能说出本章最核心的命令或操作套路
4. 是否能说明这些命令在什么场景下使用
5. 是否能说出至少一个仍不确定的问题

可记录为 `0-5` 分。

判断规则：

- `green`
  - 通常 `4/5` 或 `5/5`
  - 没有关键误解
  - 可以直接进入下一章
- `yellow`
  - 通常 `3/5`
  - 或 `4/5` 但仍有一个关键场景不顺手
  - 可以推进，但应标记 `revisit`
- `red`
  - 通常 `0/5` 到 `2/5`
  - 或存在关键误解
  - 先不要推进

## 8. 每日文章的标准结构

每日文章尽量遵循如下结构：

1. `今日主题`
2. `学习目标`
3. `前置回顾`
4. `典型场景`
5. `最小命令集`
6. `它是怎么用的`
7. `常见操作套路`
8. `环境差异：vim / nvim / LazyVim`
9. `今日练习（5-10 分钟）`
10. `今日问题与讨论`
11. `常见误区或易混点`
12. `扩展内容`
13. `今日小结`
14. `明日衔接`
15. `复习题`

### 8.1 front matter 要求

每日文章应至少包含：

- `title`
- `date`
- `lastmod`
- `tags`
- `categories`
- `slug`
- `summary`

推荐：

- `tags: [Vim, Neovim, LazyVim, Editor]`
- `categories: [工具学习]`

### 8.2 文章风格要求

- 主线强调高频、常用、能马上带来手感提升的内容。
- 不要把文章写成命令词典。
- 真实任务优先，例如：
  - 改变量名
  - 快速删改一段文本
  - 跨文件切换
  - 搜索替换
  - 项目内查找
  - 代码跳转
- 每篇都要有一个短练习，帮助形成肌肉记忆。

### 8.3 拓展内容的定位

- `扩展内容` 必须保留，但不能喧宾夺主。
- 它的作用是：
  - 留住低频但有价值的能力
  - 给以后回查提供入口
- 每篇扩展内容控制在 `3-5` 个点以内。
- 不要求当天掌握扩展内容。

### 8.4 Mermaid 的使用原则

Vim 学习不要求每篇都有图，但在以下情况可以使用 Mermaid：

- 模式切换关系
- 窗口 / buffer / tab 概念区别
- `vim -> nvim -> LazyVim` 的工作流映射
- 项目内操作流程

优先使用：

- `flowchart`
- `stateDiagram-v2`

## 9. 默认 14 天学习主线

如果索引没有更细的安排，默认按下列主线推进：

0. 总览与环境准备
1. Vim 的工作方式，不再写乱
2. 移动：从“硬挪”到“直接到位”
3. `operator + motion`
4. 文本对象
5. 搜索 / 替换 / 可视模式
6. buffer / window / split
7. 命令行模式与常用 Ex 操作
8. Neovim 的定位与和 Vim 的实际差异
9. Neovim 的终端、剪贴板、配置入口
10. LazyVim 总览与 leader 思维
11. LazyVim 文件 / 搜索 / buffer 工作流
12. LazyVim 代码导航与 LSP 基础操作
13. LazyVim 日常编辑闭环
14. 回顾、巩固与高频扩展

可在主线后追加 `扩展总览` 文章，例如：

- 宏 / 寄存器 / marks / folds / 高级替换
- LazyVim 排错与定制边界

## 10. 学习模式与问答模式

### 10.1 学习模式

当用户表达“开始学习”“继续学习”“今天学习某个主题”等意图时：

1. 先恢复进度
2. 判断应更新当天文章还是创建下一天文章
3. 为当天确定一个主主题
4. 如果涉及具体用法，优先结合：
   - 本地实际行为
   - 帮助文档
   - 官方文档
5. 生成或更新当天文章
6. 更新索引文件

### 10.2 问答模式

当用户提出 Vim / Neovim / LazyVim 相关问题时：

1. 先直接回答问题
2. 回答尽量结合：
   - 实际命令
   - 典型场景
   - 本地或官方帮助依据
3. 再把对当天学习有价值的问答内容写入当天文章的 `今日问题与讨论`
4. 如有必要，同步更新索引中的掌握状态

## 11. 回答用户时必须说明的内容

每次完成后，回复中应明确说明：

- 当前是第几天
- 当前学到哪里
- 本次新建或更新了哪些 Markdown 文件
- 本次主要参考了哪些帮助、文档或本地命令
- 下一步建议学习什么

## 12. 重要边界

- 不要假装通过对话记住学习进度，必须通过扫描已有 Markdown 恢复状态。
- 不要把索引文件写成长篇正文。
- 不要让低频技巧污染主线。
- 不要把 Vim 学习写成插件清单或炫技命令集合。
- 用户的最终目标是日常可用的 `nvim + LazyVim`，前面的 `vim` 和 `nvim` 内容都是为了打稳手感和心智模型。
