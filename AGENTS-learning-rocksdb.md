# RocksDB 学习工作流长期规则

当用户要求持续学习、讲解、问答、整理或持久化 RocksDB 相关内容时，优先遵守本文件。

本文件定义的是长期固定规则，不是某一次 session 的临时任务说明。

## 1. 核心目标

- 目标是建立一个长期、连续、可回看、可持久化的 RocksDB 学习系统。
- 每天的主产物是一篇可以长期积累和回顾的 Markdown 学习文章。
- 学习系统的第一优先级是“方便学习者回看、整理、串联知识”，不是“让 AI 做复杂学习评估”。
- 写作风格采用“源码驱动的学习型文章”。
- 内容组织以“每日文章”为中心，索引/状态只做辅助。
- 当数据结构、模块关系、调用链、状态变化或项目流程适合图示时，应优先考虑使用 Mermaid 图辅助说明。

## 2. 事实标准与资料优先级

### 2.1 本地源码是事实标准

- RocksDB 本地源码目录固定为 `D:\program\rocksdb`。
- 当前本地仓库中的 RocksDB 源码是事实标准。
- 当网络文章、Wiki、Issue、评论区说法、历史印象与当前本地源码不一致时，以当前本地源码为准。

### 2.2 外部资料的角色

- 外部资料不是事实标准，而是“问题来源”和“理解补充来源”。
- 外部资料的价值主要在于：
  - 提出值得追问的问题
  - 补充某个设计背后的动机
  - 暴露常见误解
  - 提供工程实践视角
- 对外部资料中的关键问题，最终都要回到本地源码验证。

### 2.3 推荐外部资料来源与优先级

外部资料应明确区分优先级。

#### 第一优先级：官方与仓库内直接相关资料

这些来源优先使用，因为它们最接近实现和维护者语境：

- RocksDB 官方 GitHub 仓库中的：
  - `GitHub Wiki`
  - `GitHub Issues`
  - `GitHub Discussions`
  - `Pull Requests` 中与设计、行为变化、实现原因有关的讨论
- RocksDB 官方文档
- RocksDB 官方博客或 Meta/Facebook 发布的 RocksDB 相关文章

#### 第二优先级：高质量技术问答与社区讨论

这些来源适合作为问题来源和经验补充，但必须回到本地源码验证：

- `Stack Overflow`
- `知乎`
- `linux.do`
- 高质量个人技术博客
- 其他技术论坛、评论区、读书笔记、课程讨论区

#### 第三优先级：低可信度或转述型资料

这些资料只能作为线索，不应直接作为结论依据：

- 无源码引用、无实验依据的二手总结
- 大量互相转载的文章
- 只给结论不给上下文的短帖

### 2.4 外部资料的使用顺序

使用外部资料时，按以下顺序：

1. 先记录问题，而不是直接接受结论
2. 判断这个问题是否对当前主题有学习价值
3. 回到 `D:\program\rocksdb` 中查相关源码
4. 最终把“源码验证后的结论”写入当天文章

### 2.5 外部资料的记录建议

如果某个外部问题被写入当天文章，尽量注明：

- 来源平台
  - 如 `GitHub Issue`、`GitHub Wiki`、`Stack Overflow`、`知乎`、`linux.do`
- 为什么这个问题值得记录
- 当前本地源码是否支持这个说法
- 如不完全支持，差异点在哪里

### 2.6 讲解时的约束

- 能指出源码位置时，不只讲抽象概念。
- 能说明调用链时，不只讲教科书定义。
- 如果某个判断并非直接由源码确认，而是从结构上推断，必须明确标注：
  - `这是基于当前本地源码结构的推断`

## 3. 产物目录与命名规范

### 3.1 输出目录

- 所有与本次 RocksDB 学习相关的新 Markdown 文件，都写入当前仓库的 `content/posts`。

### 3.2 文件命名

- 所有新生成的 RocksDB 学习文件统一使用此前缀：
  - `learning-rocksdb-day`
- 文件命名格式统一为：
  - `learning-rocksdb-dayXXX-YYYY-MM-DD-<topic>.md`

说明：

- `XXX` 为三位数字编号，如 `000`、`001`、`002`
- `YYYY-MM-DD` 为本地日期
- `<topic>` 为小写英文短语，使用连字符连接

### 3.3 文件类型

建议至少维护以下文件：

1. 轻量索引/状态文件
   - 固定文件名：`learning-rocksdb-day000-index.md`
2. 每日学习文章
   - 例如：`learning-rocksdb-day003-2026-04-01-write-path-and-wal.md`

默认不单独维护独立问答日志，问答内容直接写入当天的学习文章中。

## 4. 文件职责划分

### 4.1 每日学习文章是主载体

- 每日文章是知识主文档。
- 每日文章优先承载真正的学习内容，而不是仅仅记录状态。
- 用户当天提出的问题、以及网络搜集到的高价值问题，都直接写入当天文章。
- 每日文章必须做到“单独阅读也能明白今天学了什么”。

### 4.2 索引/状态文件是辅助载体

- 索引/状态文件只负责导航、进度恢复和轻量状态维护。
- 索引文件不承载长篇知识正文。
- 索引文件应保持唯一；如果 `content/posts` 中已存在 `learning-rocksdb-day000-index.md`，则一律更新它，不再创建新的索引文件。
- 索引文件的目的是让新 session 能快速恢复：
  - 当前学到哪里
  - 哪些主题完成了
  - 哪些主题只是部分掌握
  - 下一步该继续什么

### 4.3 不要过度拆散

- 不要把知识内容拆到多个分散文件里，导致学习者回看困难。
- 对用户来说，最重要的是“每天那篇文章里写清楚学了什么”。
- 对 AI 来说，索引文件只需要保存足够恢复上下文的信息，不需要变成复杂管理系统。

## 5. 新 Session 的恢复流程

每次开始前，必须按以下顺序工作：

1. 扫描 `content/posts` 中所有 `learning-rocksdb-day*.md`
2. 补充扫描历史上其他标题、文件名或正文中包含 `RocksDB`、`rocksdb` 的旧文章，作为背景参考
3. 识别并读取索引文件
4. 根据索引文件和最近的每日文章，恢复：
   - 当前是第几天
   - 已经学过哪些主题
   - 哪些主题是 `done`
   - 哪些主题是 `revisit`
   - 最近一次学习的主线停在哪
   - 目前的薄弱点是什么
5. 如果索引文件与每日文章冲突，以已有每日文章和实际文件序列为准，修正索引
6. 若本次任务涉及具体实现细节，先读取 `D:\program\rocksdb` 中相关源码，再组织回答或生成新文章

## 6. 索引/状态文件应记录什么

索引文件应保持“轻量但够用”。

至少记录以下信息：

- 当前学习总天数
- 当前最近一次学习的主主题
- 每天文章索引：
  - day 编号
  - 日期
  - 主题
  - 文件名
- 每天的轻量状态：
  - `next`
  - `revisit`
  - `done`
- 下一步建议学习什么
- 当前仍需补看的关键点

为了让新 session 不只知道“学过什么”，还知道“学得怎么样”，每一天还应保留一份轻量掌握快照，建议字段如下：

- `understanding_status`
  - 取值：`green` / `yellow` / `red`
- `mastery_score`
  - 取值：`0-5`
- `weak_points`
  - 当前仍不稳定的 1 到 3 个点
- `source_anchors`
  - 本章真正读过的关键源码文件

以下字段可选，仅在确实有必要时再记录：

- `ready_for_next`
  - 取值：`yes` / `no`
- `next_review_trigger`
  - 后续学到什么主题时需要回头复习

说明：

- 这些状态字段应写在索引文件中，而不是让它们淹没每日文章正文。
- 状态是为了帮助恢复上下文，不是为了给学习者制造负担。

## 7. 如何判断一章是否可以进入下一章

这是本工作流中的硬规则。

### 7.1 总原则

- 不要求 100% 吃透后再推进。
- 采用“理解约 80% 即可推进”的原则。
- 但不能在存在关键误解的情况下推进。

### 7.2 四个核心通过条件

判断某章是否可推进时，至少检查以下四项：

1. `源码定位`
   - 能说出这一章对应的核心目录、文件、类、函数在哪里
2. `机制理解`
   - 能用自己的话讲清楚它解决什么问题、主流程怎么走、和上下游怎么衔接
3. `关键问答`
   - 能回答 3 到 5 个本章核心问题，而不是只会复述名词
4. `边界感`
   - 能说清楚它与前后章节的边界，不把邻近概念混为一谈

### 7.3 五项检查表

更具体地说，每章结束时，都应做一个 5 项过关检查：

1. 我能指出本章主要源码文件
2. 我能描述主流程
3. 我能说出 2 到 3 个关键数据结构
4. 我能回答本章复习题
5. 我能说出至少一个仍不确定的问题

每项可记 1 分，总分为 5 分。

### 7.4 通过判定规则

- `green`
  - 通常对应 `4/5` 或 `5/5`
  - 且没有关键误解
  - 可以直接进入下一章
- `yellow`
  - 通常对应 `3/5`
  - 或者 `4/5` 但仍有一个关键链路没有彻底闭环
  - 可以进入下一章，但必须在索引中记录薄弱点，并标记后续回看条件
- `red`
  - 通常对应 `0/5`、`1/5`、`2/5`
  - 或虽然分数不低，但存在明显关键误解
  - 不应推进，先补充学习或问答澄清

### 7.5 状态的使用方式

- 掌握状态主要写入索引文件。
- 每日文章末尾可以带一个简短“本章结论”摘要，但不要让状态信息压过正文。
- AI 的评估只是辅助；真正重要的是：
  - 当天文章是否把知识讲清楚了
  - 新 session 能否从索引中恢复“这章掌握到什么程度”

## 8. 每日文章的标准结构

每日文章应尽量遵循以下结构。文章可以按复杂度伸缩，但主框架应稳定。

1. `今日主题`
2. `学习目标`
3. `前置回顾`
4. `源码入口`
5. `它解决什么问题`
6. `它是怎么工作的`
7. `关键数据结构与实现点`
8. `源码细读`
9. `今日问题与讨论`
10. `常见误区或易混点`
11. `设计动机`（可选）
12. `横向对比`（可选）
13. `工程启发`（可选）
14. `今日小结`
15. `明日衔接`
16. `复习题`

### 8.1 front matter 要求

每日文章应包含 front matter，至少包括：

- `title`
- `date`
- `lastmod`
- `tags`
- `categories`
- `slug`
- `summary`

推荐：

- `tags: [RocksDB, Database, Storage]`
- `categories: [数据库]`

### 8.2 每日文章的定位

- 每日文章不是流水账。
- 每日文章也不是纯对外博客。
- 它应当是“结构完整、可连续阅读的学习型文章”。
- 风格上：
  - 70% 学习笔记
  - 30% 博客文章

### 8.2.1 三个固定可选补充视角

以下三类内容不单独构成学习主线，但应作为主线文章的固定可选补充模块：

- `设计动机`
- `横向对比`
- `工程启发`

它们的定位是：

- 主线内容负责回答：
  - 这是什么
  - 它怎么工作
  - 源码在哪里
- 这三个补充视角负责回答：
  - 为什么 RocksDB 这样设计
  - 其他系统在类似问题上怎么做
  - 这部分实现有哪些值得借鉴的工程思想

明确规则：

- 不把这三类内容单独提升为新的学习主线
- 也不要随口零碎带过
- 而是作为“主线文章中的结构化补充小节”使用

### 8.2.2 何时补充这些小节

当当前主题满足以下任一条件时，至少补其中 1 个小节：

- 存在明显设计取舍
  - 例如 `LSM-Tree`、`Compaction`、`Write Stall`、`Block Cache`
- 与其他数据库或存储系统的做法差异明显
  - 例如 `LSM` 与 `B+Tree`
  - `Block Cache` 与传统数据库 `Buffer Pool`
  - RocksDB 的 `Snapshot` 与其他 MVCC 实现
- 有明确工程借鉴价值
  - 例如 `WAL`
  - `VersionSet`
  - 后台线程调度
  - 参数化设计与调优接口

当当前主题只是局部源码细节时，可以省略这些小节，例如：

- 某个函数内部的小逻辑
- 某个结构体字段的局部解释
- 某段实现的边角行为

### 8.2.3 三个补充小节的写法要求

#### `设计动机`

适合回答：

- 为什么要这样设计
- 这个设计主要优化了什么
- 它接受了哪些代价

常见适用主题：

- `LSM-Tree`
- `WAL`
- `Compaction`
- `Write Stall`
- `Column Family`

#### `横向对比`

适合回答：

- 其他系统在类似问题上常见怎么做
- RocksDB 的做法和它们相比有什么不同
- 这种差异会带来什么行为差别

常见适用主题：

- `LSM-Tree` 对比 `B+Tree`
- `Block Cache` 对比传统数据库 `Buffer Pool`
- RocksDB 的 `Snapshot` 对比其他数据库的 MVCC 可见性处理
- 不同存储系统在 `Compaction`、缓存、事务支持上的差异

使用要求：

- 对比只服务于理解当前主题
- 不做泛泛的数据库综述
- 不因为横向扩展而打断主线

#### `工程启发`

适合回答：

- 这部分实现里有哪些值得迁移到其他项目的思想
- 哪些边界划分、接口设计、后台调度、缓存策略值得借鉴

常见适用主题：

- `WAL + MemTable + 后台整理` 的职责拆分
- `Version / Manifest` 这样的元数据管理方式
- `Bloom Filter + Cache` 的读优化组合
- 调参接口与限流、stall 等保护机制

### 8.2.4 放置位置

这三个补充小节应放在主线内容之后、总结之前。

推荐顺序为：

1. 主线正文
   - `源码入口`
   - `它解决什么问题`
   - `它是怎么工作的`
   - `关键数据结构与实现点`
   - `源码细读`
   - `今日问题与讨论`
   - `常见误区或易混点`
2. 可选补充小节
   - `设计动机`
   - `横向对比`
   - `工程启发`
3. 收尾部分
   - `今日小结`
   - `明日衔接`
   - `复习题`

### 8.2.5 强度控制

- 每篇文章不要求三个都写。
- 一般补 1 到 2 个就够。
- 如果主题特别适合，也可以三个都补。
- 不要把每篇文章都写成数据库比较论文。

### 8.3 Mermaid 图示要求

每日文章在以下场景下，应优先判断是否使用 Mermaid 图：

- 数据结构关系
  - 例如 `SkipList`、`MemTable`、`VersionSet`、`BlockBasedTable` 的结构关系
- 模块关系
  - 例如 `DBImpl`、`ColumnFamily`、`VersionSet`、`TableReader` 之间的协作关系
- 调用链或流程
  - 例如 `Write Path`、`Read Path`、`Flush`、`Compaction`、恢复流程
- 状态变化
  - 例如 MemTable 从 mutable 到 immutable，再到 flush 的过程
- 生命周期
  - 例如 WAL、SST、Manifest 的生成、使用、回收过程

使用 Mermaid 的原则：

- 图应服务于理解，不是装饰。
- 当纯文字很难快速说清结构或流程时，优先补 Mermaid 图。
- 图和正文必须互相对应，不能只放图不解释。
- 图尽量简洁，只保留当前主题真正关键的节点和边。
- 如果某个流程跨越过大，不要把整套系统塞进一张图里，应拆成多张小图。

推荐优先使用的 Mermaid 类型：

- `flowchart`
  - 用于流程图、模块协作、主线调用关系
- `sequenceDiagram`
  - 用于请求时序、读写路径、模块交互顺序
- `classDiagram`
  - 用于类、结构体、对象关系
- `stateDiagram-v2`
  - 用于状态迁移

使用规则：

- 每天文章不要求必须有图，但当主题明显适合图示时，应主动补图。
- 如果当天主题包含“结构”或“流程”两类核心内容，至少应考虑补 1 张 Mermaid 图。
- Mermaid 图应直接写入 Markdown 中，方便文章持久化和后续回看。

## 8.4 默认学习路径

如果索引文件没有给出更精确的下一步，默认按下面这条主线推进。

这条主线优先吸收以下要求：

- 核心架构必须先建立整体图景
- 写路径与读路径必须作为主线反复回到源码中串起来
- `Compaction`、`MemTable`、`Block Cache`、`Bloom Filter`、`Column Family`、事务并发、配置调优和高级特性都要纳入正式学习路径
- 需要单独补入“磁盘管理相关”，包括：
  - RocksDB 如何读写磁盘
  - 文件 I/O 相关抽象
  - 数据从内存到磁盘、从磁盘回到内存的路径
  - 缓冲与缓存是如何配合处理的
- 磁盘管理相关内容应放在 `SST` 与 `Read Path` 之后，再系统展开，避免过早引入过多实现细节

### 第一阶段：核心架构总览（必看）

1. RocksDB 整体架构与 LSM-Tree
   - 理解 `MemTable -> Immutable MemTable -> SST` 的基本演进路径
2. DB 打开流程与核心对象关系
   - 例如 `DBImpl`、`ColumnFamily`、`VersionSet`、`TableCache`
3. WAL 的作用与崩溃恢复基本模型
4. SST 文件格式总览
   - `Data Block`
   - `Index Block`
   - `Filter Block`
   - `Meta Block`
5. Manifest 与 Version 总览
   - RocksDB 如何管理 SST 元信息和版本快照

### 第二阶段：写路径主线（必看）

6. 写路径总览
   - `Put -> WAL -> MemTable -> Flush -> Compaction`
7. `WriteBatch`
   - 原子批量写入的实现
8. `WriteGroup` 与写入合并优化

## 9. 当天文章中的问题如何写

### 9.1 问题都进入当天文章

以下两类问题都直接写入当天文章的 `今日问题与讨论` 部分：

- 用户当天提出的问题
- 外部高价值问题
  - 例如 Wiki、Issue、博客、评论区、论坛中的问题

默认不单独维护独立问答日志。

### 9.2 问题章节的内部结构

建议在 `今日问题与讨论` 下分为两个小节：

- `我的问题`
- `外部高价值问题`

### 9.3 用户问题的记录模板

每个用户问题建议至少包含：

- `问题`
- `简答`
- `源码依据`
- `当前结论`
- `是否需要后续回看`

### 9.4 外部问题的记录模板

每个外部问题建议至少包含：

- `问题`
- `来源类型`
  - 如 `wiki`、`issue`、`blog`、`comment`、`forum`
- `为什么这个问题重要`
- `源码依据`
- `当前结论`
- `是否与当前本地源码一致`

### 9.5 外部问题的筛选原则

并非所有外部问题都应写入当天文章。

优先收录以下问题：

- 能帮助理解当天主题
- 能暴露常见误区
- 能让源码里的某段设计变得更清楚
- 能为后续章节做铺垫

不符合这些条件的问题可以不写入，避免污染正文。

### 9.6 跨章节大问题

对于跨多天的大问题，例如：

- `sequence number` 与 `snapshot`
- `VersionSet` 与 `compaction`
- `read path` 与 `iterator`

当天文章只写“与当天主题最相关的切面”，不要强行一次讲完整个专题。

## 10. 学习模式与问答模式

### 10.1 学习模式

当用户表达“开始今天的学习”“继续学习”“继续下一天”等意图时：

1. 先恢复进度
2. 判断今天是：
   - 更新当天文章
   - 还是创建下一天文章
3. 为当天确定一个主主题，必要时允许一个副主题，但不要散
4. 先查相关源码，再生成文章
5. 生成主线内容后，主动检查是否需要补：
   - `设计动机`
   - `横向对比`
   - `工程启发`
6. 更新索引文件

### 10.2 问答模式

当用户提出 RocksDB 相关问题时：

1. 先回答问题
2. 回答时优先结合本地源码
3. 再把对当天学习有价值的问答内容写入当天文章
4. 如问答改变了对某章掌握状态的判断，可同步更新索引中的状态

## 13. 重要边界

- 不要假装通过对话记住学习进度，必须通过扫描已有 Markdown 恢复状态。
- 不要把源码目录和文章持久化目录混淆。
- 不要只回答不落盘；凡是与当天学习直接相关的重要内容，都应写入当天文章。
- 不要让索引文件承载长篇知识正文。
- 不要把文章拆得太散，导致回看困难。
- 每日文章是主载体，索引文件是导航和恢复工具。
